## The Content Taxonomy — the operational spec every Pylon media SKU delivers

This is the deliverable spec that makes V2's Team Media SKUs concrete. Encode it in
`skills/content-taxonomy/SKILL.md` in pylon-os.

### Content 1 — Player POV Reel (per event, per selected star athlete)

- **Format:** Instagram Reel
- **Shot list:**
  - Warm-up shots (mic'd) — enough to build anticipation
  - In-game action — high-energy skill highlights
  - Sideline conversations — coach + teammate interactions
  - "Couch conversations" (if allowed / adds value)
  - Post-game — player talking to camera, post-win energy, teammate interactions
  - Long single-shot sequences (full plays) — reusable for post + carousel
- **Voice:** Documentary. Player-narrated OR silent + captioned.
- **Visual direction:** simple editing (let athlete tell story), color grade optional, action photos of player for reel thumbnail, cinematic realism
- **Deliverable:** 1 finished reel + 1 thumbnail card + 3–5 long single-shot clips for downstream repurposing
- **Purpose:** showcase athlete's skill, personality, leadership. Powers Pylon Athlete Transcript (V2 SKU #9) subscription conversion.

### Content 2 — Action Photos Carousel (per event, per team)

- **Format:** Instagram Post, Carousel
- **Shot list:**
  - Filmer/shooter choice, usable across multiple media formats
  - Focus on main athlete(s) for narrative
  - Limit workload — don't shoot everything, shoot the moments
- **Visual direction:** cinematic realism, shallow DOF, high-contrast edits, candid emotion, dynamic body positioning (mid-air, full stride, intense expressions), clean backgrounds when possible
- **Deliverable:** 10-slide branded carousel + full-res photo pack for downstream use
- **Purpose:** evergreen general-use content. Powers ambassador program amplification + Team Media Package (V2 SKU #1a/#1b) deliverable requirement.

### Content 3 — Support Content (per event, per team)

Three variants per event:

**3a. Team Highlight**
- Mix of star player shots + team + coach shots
- Additional b-roll for production value (detail shots, close-ups, wide shots — crowd, equipment, small actions)
- Shows team chemistry

**3b. Single Shots (30-second clips)**
- 30-sec long clips filmed intentionally for single-shot use
- Coaches speech, last-minute plays, intuitive captures
- Personal, unedited feel — not overproduced

**3c. Individual Player Focus**
- 5 photos/videos of specific player being spotlighted
- "Player focus" pivot from the team-first content

- **Deliverable:** 3 support pieces (team highlight + single shots + individual focus)
- **Purpose:** cohesive supporting content that adds page value without shoot-load spike. Feeds Sunday Salvo, ambassador posts, Pylon Extended Rankings pages.

### The 6-piece minimum

Every event produces AT LEAST:
- 1 × Content 1 (POV Reel)
- 1 × Content 2 (Action Photos Carousel)
- 3 × Content 3 (support pieces)
- 1 × Team Highlight (if bundled)

Six pieces per team per event = the concrete deliverable spec for **Pylon Team Recap** ($299/event, V2 SKU #1a). Season Pack (V2 SKU #1b, $1,499) × 5+ events = 30+ pieces per team per season.

### The 25% OFF referral loop mechanic

Codify in the tojcampaign build. New table:

```sql
create table pylon_creator_referrals (
  id uuid primary key default gen_random_uuid(),
  creator_email text not null,
  creator_name text not null,
  referral_code text unique not null,             -- e.g. 'JMILLER25'
  discount_percent int not null default 25,
  active_at_event_id uuid references pylon_events(id) on delete set null,
  active_from timestamptz not null default now(),
  active_until timestamptz,
  uses int default 0,
  tips_earned_cents int default 0,
  status text default 'active',
  created_at timestamptz not null default now()
);

create table pylon_referral_conversions (
  id uuid primary key default gen_random_uuid(),
  referral_id uuid not null references pylon_creator_referrals(id),
  package_order_id uuid not null references pylon_package_orders(id),
  attributed_at timestamptz not null default now()
);
```

Route: `POST /api/pylon/creator-referrals` to mint a new code per creator per
event. Applied at checkout via URL parameter or discount code entry.

## Before / During / After event workflow

The proven Subject Media workflow, codified as
`skills/event-week-runbook/SKILL.md` in pylon-os. All three phases delivered by
the Pylon Media pod (with TOJ system support).

### Before the event

- Create a QR code for the Pylon Media landing page (`/pylon-player-packages.html`) event-scoped so package orders auto-associate with the correct event
- Design a graphic announcing Pylon Media coverage at the event, embedding the QR code
- Assign creators to specific athletes/teams for the event (via `/admin/pylon-media-assignments.html`)
- Send pre-event brief to each assigned creator with content taxonomy spec + visual direction rubric + team/athlete backgrounder

### During the event

- Vendor booth at the event with Pylon Media samples on display
- Sell packages on behalf of Pylon at the booth (creator or dedicated seller runs booth)
- Package purchases upload to the Pylon Media Dashboard (`/admin/pylon-media.html` in tojcampaign, or dash.xpandsports.com if we integrate — see below)
- Content creators are auto-assigned to purchased packages via the assignment logic
- Live event posting (V2 SKU #10) runs in parallel

### After the event

- DM the athlete/parent through Instagram with a sample photo/video from their package
- Post sample pictures publicly to drive attention (with creator credit tagged, drives Ambassador/creator distribution)
- Contact teams and event staff for follow-on relationships (Season Pack upsell, City League host applications, sponsor introductions)

### Skill file structure

`skills/event-week-runbook/SKILL.md` has three sections (Before / During / After),
each with a checklist + timing (T-14 → T-0 → T+7) + who owns what.

## Xpand Sports infrastructure — integrate first, migrate later

Kyron's existing Subject Media infrastructure is live at:
- **`subjectmedia.xpandsports.com`** — public landing page with $69/$99/$199 tiers
- **`dash.xpandsports.com`** — internal CSV/revenue dashboard
- **`z4otpbmnpouvwt6orgni.app.client…`** — Subject Media "About" client-app URL

For V2.8, do **NOT** rebuild this from scratch inside tojcampaign. Two options:

### Option A (recommended) — integrate

- Keep `dash.xpandsports.com` as the operational dashboard for the pod's day-to-day operations (creator assignments, order routing, content delivery status)
- Pylon-brand a new landing page inside tojcampaign at `/pylon-player-packages.html` for player-level SKU checkout, which posts orders to the same backend (or via a webhook to dash.xpandsports.com if it has an intake API)
- `pylon_package_orders` (V2 schema) records the order; a webhook or scheduled sync forwards it to Xpand Sports for creator-crew delivery routing
- Faster to ship, respects existing infrastructure

### Option B — rebuild in tojcampaign

- Full migration of Xpand Sports functionality into tojcampaign (`/admin/pylon-media.html`)
- Retire xpandsports.com subdomains for Pylon use (keep them alive for TOJ's Subject Media proof-of-concept brand per V2.7)
- Slower, but everything sits in one repo

**Recommendation: Option A for V2.8**, migrate to Option B in V3 (2027 season)
once volume justifies the consolidation cost. Ship velocity matters more than
architectural purity in year one.

The other Claude Code should ask Kyron A vs B at build time. If unanswered,
default to A (integrate).
